# SPEC02 — 错误处理与安全退出

> 从 SPEC01 拆分，第一阶段暂不实现  
> 创建日期: 2026-04-17

---

## 1. 目标

为 Agent Loop 提供完善的错误恢复和安全退出机制，确保各类异常场景下系统行为可控、用户体验友好。

---

## 2. 错误处理策略

| 场景 | 处理方式 |
|------|---------|
| API 网络错误 / 5xx | 指数退避重试，最多 3 次，间隔 1s/2s/4s |
| API 4xx (key 无效等) | 直接报错退出，提示用户检查配置 |
| API 返回格式异常 | 记录原始响应，返回友好错误信息 |
| 工具调用参数解析失败 | 返回错误信息给 LLM，让其修正 |
| 工具执行异常 | 捕获异常，返回错误文本给 LLM |
| Bash 命令超时 | 返回超时提示给 LLM |
| 达到最大迭代次数 | 终止循环，提示用户 |
| 输出过长 (token 爆炸) | 截断工具输出至 50000 字符 |

---

## 3. 错误恢复

当工具执行失败时，系统应将错误信息以结构化文本回传给 LLM，使其能够：

1. **理解失败原因** — 错误信息应包含：工具名、错误类型、错误摘要
2. **自主调整策略** — LLM 根据错误信息决定重试、换方案或告知用户
3. **避免死循环** — 同一工具连续失败超过 3 次时，在错误信息中追加提示，引导 LLM 放弃该路径

错误回传格式：

```
[Tool Error] tool=bash, type=TimeoutExpired
Command timed out after 120s.
Suggestion: try a simpler command or break the task into smaller steps.
```

---

## 4. 安全退出

### 4.1 迭代上限

- 默认 `max_iterations = 50`
- 达到上限时返回提示信息，不抛异常
- 提示内容：`[Agent reached maximum iteration limit (50). Please continue with a new query.]`

### 4.2 用户中断 (Ctrl+C)

- CLI 层捕获 `KeyboardInterrupt`
- 第一次 Ctrl+C：中断当前 LLM 调用或工具执行，返回到 REPL 提示符
- 第二次连续 Ctrl+C：退出程序
- 中断时保留已有消息历史，用户可继续对话

### 4.3 API 重试耗尽

- 3 次重试均失败后，向用户展示最后一次错误
- 保留会话状态，用户可重新输入触发重试

---

## 5. 验收标准

1. **错误恢复**：工具执行失败时，LLM 能收到错误信息并调整策略
2. **安全退出**：迭代次数超限时优雅终止；用户可 Ctrl+C 中断
3. **重试机制**：API 瞬时故障后能自动恢复
4. **无死循环**：同一错误不会导致无限重试
